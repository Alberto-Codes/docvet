## Successes

- **LSP-as-architectural-precedent eliminated design risk** — Having `lsp.py` as a template meant the MCP server was adapted, not designed from scratch. Module-level server instance, single public `start_server()`, internal helpers — all reused patterns. Zero-debug across all 3 stories.
- **Code review caught real defects** — 29.1 code review found a directory scoping bug (H2: `DiscoveryMode.ALL` ignoring target directory) and an O(n*20) performance issue (M1: linear rule lookup replaced with `_RULE_TO_CHECK` dict). These are production bugs, not style nits.
- **Integration tests with MCP client SDK** — 29.2 used the `mcp` package's `stdio_client` and `ClientSession` to spawn a real server process, call tools, and verify responses end-to-end. The parity test comparing MCP output against CLI JSON output is a contract test that catches drift.
- **Release-please + OIDC CI automation** — `server.json` version fields wired into release-please `extra-files` (29.3), plus automated MCP registry publish via OIDC in the CI pipeline (post-epic). Zero manual steps on release.
- **Real-world adoption** — Colleagues set up docvet MCP at work, validating the docs, registry publishing, and developer experience end-to-end.
- **+64 net new tests** — 45 MCP unit tests (29.1) + 4 CLI unit tests (29.2) + 6 MCP integration tests (29.2) + others = 64 net new.

---

## Challenges

- **Story file completeness lagging behind implementation** — 29.1 story file was completely blank (all tasks unchecked, Dev Agent Record empty, File List missing) when code review began. Same pattern as 28.2. Recurring issue that needs structural enforcement.
- **Environment isolation in integration tests** — 29.2 parity test initially picked up system-installed docvet v1.0.0 instead of venv v1.7.0. Fixed by using `sys.executable` parent path to locate the venv's console script. Environment assumptions are fragile.
- **Python namespace shadowing** — `mcp()` CLI function name shadows the `mcp` package, requiring lazy import and `start_server as mcp_start_server` alias. Namespace collisions are a Python gotcha with no great solution.

---

## Key Insights

1. **Architectural precedent is the highest-leverage reuse pattern** — LSP → MCP eliminated all design risk and produced zero-debug implementations across 3 stories. When building a new protocol integration, find an existing one to adapt.
2. **Code review catches real defects** — Directory scoping bug and O(n*20) performance issue found and fixed before merge. Code review is not ceremony; it's the last line of defense.
3. **CI automation compounds** — Release-please version bumps + OIDC registry publish = zero manual steps forever. Every manual step automated is a future mistake prevented.
4. **Real-world adoption is the ultimate validation** — Colleagues setting up docvet MCP at work confirms the docs and developer experience work. No amount of testing replaces real users.
