## Challenges

### Critical Finding: Copilot caught 8 bugs our process missed

Across 4 PRs, GitHub Copilot's automated review found 8 issues that passed through detailed story specs, Claude Opus 4.6 implementation, internal code review, AND party-mode consensus:

| Root Cause | Findings | Severity |
|-----------|----------|----------|
| AST body includes docstring node | #3: `_is_stub_function` broken for documented functions | HIGH — primary use case broken |
| Cross-rule interaction | #7: missing-docstring vs overload-has-docstring impossible state | HIGH — user-facing conflict |
| Regex `\s` matches newlines | #1: sphinx directive, #5: numpy underline cross-line | MEDIUM — silent misbehavior |
| Test helper selects wrong target | #4: class symbol not method, #8: filtered findings hide interaction | MEDIUM — false-green tests |
| Conditional skip not propagated | #2: phantom griffe metric in --summary | LOW — cosmetic |

**Root cause analysis:** All 8 findings share a pattern — the code does exactly what the developer intended, but the developer's mental model had incorrect assumptions. The current rules files are 80% style guidance (which pre-commit enforces) and 0% domain-specific semantic guidance (which only the LLM can catch).

## Key Insights

1. **Attention budget matters — shift from style to semantics.** The LLM reads ~186 lines of python rules saying "use 4 spaces" and "use f-strings" — ruff catches all of this. Zero lines about AST body counting with docstrings or `\s` matching newlines. Every line of ruff-duplicate guidance is a line of domain knowledge deprioritized.

2. **Our code review reviews implementation against intent, not assumptions against reality.** All 8 Copilot findings were correct implementations of incorrect assumptions. The spec for 34.2 said "check `len(body) == 1`" — it baked the assumption in. Code review confirmed the code matched the spec. Copilot reviewed the diff against the full codebase and saw the interaction.

3. **Tests can lie — test validity is a distinct concern from test coverage.** Finding #4 (helper selects wrong symbol) and #8 (filtered findings hide conflicts) are tests that pass for wrong reasons. Our dev-quality-checklist covers assertion strength but not test validity.

4. **Cross-rule interactions are an architectural blind spot.** Adding `overload-has-docstring` to the same module as `missing-docstring` created an impossible state for `@overload` symbols. Story specs describe individual rules, not how they interact with existing rules.

5. **Module size pressure is building.** `enrichment.py` now has section parsing, sphinx detection, numpy detection, 10+ check functions, the dispatch table, and 5+ helper functions. Epic 35 adds 6 more rules. A module size trigger rule is needed.

## Action Items (Unanimous Consensus)

### Process Improvements

1. **Remove `.github/instructions/` files**
   - Delete `python.instructions.md` and `pytest.instructions.md`
   - `.claude/rules/` is the single source of truth for Claude Code
   - Owner: Dev agent, next story

2. **Rebalance `.claude/rules/python.md`**
   - Trim style rules ruff/ty already enforce (imports, formatting, naming, line length, quotes)
   - Keep semantic rules (no mutable defaults, no bare exceptions, `from __future__`)
   - Add reference: "See project-context.md for AST and regex domain patterns"
   - Owner: Dev agent, next story

3. **Add test validity principles to `.claude/rules/pytest.md`**
   - "Verify test helpers select the intended symbol type — confirm `kind` and `name` before asserting"
   - "Test unfiltered findings when adding rules to modules with existing rules"
   - "Verify skip paths are reachable — confirm fixture actually triggers the intended guard clause"
   - Owner: Dev agent, next story

4. **Add test validity section to `.claude/rules/dev-quality-checklist.md`**
   - Same three principles as Action 3, framed as checklist items for during-development use
   - Owner: Dev agent, next story

5. **Overhaul `project-context.md`**
   - Fix stale content: remove sphinx/numpy prohibition, fix `develop` → `main`, update date
   - Add "Domain Implementation Patterns" section:
     - AST: body includes docstring node, `ast.get_docstring()` vs `node.body`, scope-aware walking vs `ast.walk()`, decorator detection (Name + Attribute forms)
     - Regex: `\s` matches `\n` (use `[^\S\n]`), sort alternatives by length descending, Google vs Sphinx parsing strategies, boundary detection consistency
     - Cross-rule interactions: verify no impossible states, presence skip propagation, enrichment dispatch considerations
     - CLI: skip propagation to all downstream consumers (stats, summary, reports, check_counts, exit codes)
   - Trim duplicated style rules
   - Owner: Dev agent, next story

6. **Story creation — add "Interaction Risk" one-liner**
   - When SM creates a story for a rule in an existing check module, include one sentence identifying existing rules and symbol overlap to verify
   - Owner: SM agent, standard practice going forward

7. **Add module size guidance to `.claude/rules/python.md`**
   - Trigger: When a source module exceeds ~500 lines or contains 3+ distinct concerns, consider extracting into a sub-package
   - Pattern: `checks/enrichment.py` → `checks/enrichment/` with `__init__.py` re-exporting, internal modules for concerns
   - Same applies to test files (ongoing splitting already in progress)
   - Owner: Dev agent, applied when natural (not forced refactor)

### Team Agreements

- Pre-commit handles style; LLM handles semantics — rule files must reflect this division
- `.claude/rules/` is the single source of truth (no `.github/instructions/` duplication)
- Party-mode consensus remains the standard for design decisions
- Test validity (not just coverage) is a first-class quality concern
- Module size triggers apply to both source and test files
